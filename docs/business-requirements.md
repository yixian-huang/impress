# 业务需求文档

## 1. 项目背景

- **项目名称**：Inkless CMS（Inkless CMS）
- **背景问题**：
  - 官网内容目前依赖代码内硬编码与分散的 i18n 键值，无法由业务人员在后台直接编辑。
  - 缺乏发布与回滚机制，内容变更需走发版流程，风险高、周期长。
  - 中英双语内容缺少统一的翻译状态与发布前校验，易出现漏翻或过期译文上线。
- **业务目标**：
  - 前台页面内容由「发布态配置」驱动，支持 zh/en 双语展示。
  - 后台支持草稿编辑、校验、发布、版本回滚，并具备双语门禁（missing/stale/done）。
  - 在保留现有 React 官网体验的前提下，逐步将内容管理能力从「代码发版」迁移到「配置发布」。

## 2. 用户与角色

- **角色清单**：
  - **访客（匿名）**：浏览官网前台，按当前语言查看已发布内容。
  - **编辑（editor）**：登录后台，编辑页面草稿、触发校验、查看版本列表与版本详情；不能发布或回滚。
  - **管理员（admin）**：在编辑权限基础上，可执行发布、回滚，以及全部后台内容管理操作。

- **每个角色的核心诉求**：
  - 访客：多语言浏览、信息准确、加载稳定。
  - 编辑：安全编辑草稿、实时看到校验与翻译状态、避免与他人编辑冲突。
  - 管理员：控制发布节奏、在出问题时快速回滚到历史版本、审计谁在何时做了何种变更。

- **权限边界**：
  - 公共读：仅可读取 `published_config`（通过 `GET /public/content/{pageKey}?locale=zh|en`），禁止透出草稿。
  - 后台写：仅操作 `draft_config`；发布/回滚时由服务端将草稿或历史快照写入发布态。
  - 发布与回滚：仅 `admin` 可调用 `POST .../publish`、`POST .../rollback/{version}`。
  - 编辑与校验：`admin` 与 `editor` 均可调用草稿的 GET/PUT、validate、版本列表/详情。
  - 审计信息：变更记录包含 `changeNote`、`operator`、`createdAt`（ISO-8601 UTC）。

## 3. 核心业务流程

- **主流程**：
  1. 编辑/管理员登录后台 → 选择页面 → 进入草稿编辑（双语同屏）。
  2. 保存草稿（PUT draft，带 `If-Match`/版本号）→ 可选：调用校验接口查看 missing/stale。
  3. 管理员确认无误后发起发布（POST publish，带 `expectedDraftVersion`）→ 服务端校验 schema 与翻译状态，通过则写 `content_versions` 并更新 `published_config`。
  4. 前台通过公共接口按 `locale` 拉取发布态配置并渲染页面。
  5. 若需回滚：管理员选择历史版本 → POST rollback → 从该版本快照生成新发布版本，不覆盖旧版本记录。

- **异常流程**：
  - 草稿并发冲突：两人同时保存同一页面草稿时，后提交者收到 `409 CONFLICT_VERSION`，需刷新后基于最新版本再编辑。
  - 发布门禁失败：存在必填字段 missing 或 stale 时，发布返回 `422 VALIDATION_FAILED`，需在后台补齐或确认翻译后再发布。
  - 未登录/无权限：接口返回 `401 UNAUTHORIZED` 或 `403 FORBIDDEN`，前端引导登录或提示权限不足。

- **边界条件**：
  - 发布时必须校验 `expectedDraftVersion`，防止「先校验后被他人改写再误发布」。
  - 回滚生成新版本号，历史版本记录保留可查。
  - 公共接口仅返回发布态，任何情况下不返回草稿；前台运行时若某 locale 缺失可回退到 zh 兜底（不改变发布校验规则）。

## 4. 关键功能需求

- **必须交付（P0）**：
  - 认证：登录、刷新 Token、登出、当前用户信息（JWT access + refresh）。
  - 后台内容：按 pageKey 获取/更新草稿（含并发控制）、校验接口（返回 valid/errors/translationStatus）。
  - 发布与回滚：发布（事务内校验 + 写版本 + 更新发布态）、版本列表与版本详情、回滚到指定版本并生成新发布版本。
  - 公共内容接口：按 pageKey + locale 返回发布态 config，供前台消费。
  - 后台管理界面：登录与会话、页面列表、草稿编辑器（双语同屏）、校验结果与翻译状态展示、发布与回滚操作（admin）。
  - 前台配置驱动：目标页面（见下）由 Public API 的 config + locale 驱动渲染，保留 i18n 兜底。

- **重要交付（P1）**：
  - 页面范围全覆盖：`home`、`about`、`advantages`、`core-services`、`cases`、`experts`、`contact`、`global`（共 8 个 pageKey）的 schema、草稿编辑与发布流程打通。
  - 双语状态规则：missing/stale/done 计算与展示，发布前门禁拦截 missing/stale 必填项。
  - 可观测性：发布成功率、校验失败率、回滚耗时、公共接口 P95 延迟等核心指标可采集。

- **可延期（P2）**：
  - 404 页面配置化。
  - 对象存储签名上传（当前配置中仅存 URL，静态资源仍走现有目录/CDN）。
  - 复杂工作流审批（多级审核、会签等）。
  - 生产环境 SQLite → PostgreSQL 的迁移与调优（开发阶段可用 SQLite）。

## 5. 非功能需求

- **性能**：
  - 公共内容接口需支持按 pageKey + locale 高效读取（单次查询发布态）。
  - 发布与回滚在事务内完成，避免长时间锁表；可观测 P95 延迟。
  - 前台首屏与路由切换依赖现有 React + 静态资源策略，不因接入配置接口明显劣化。

- **安全**：
  - 认证使用 JWT（access + refresh），refresh 可落库作废。
  - 后台接口全部鉴权，按角色做发布/回滚与编辑权限隔离。
  - 公共接口仅返回发布态数据，禁止将草稿、版本快照或内部错误细节暴露给未认证用户。

- **可用性**：
  - 开发环境可快速启动（SQLite + 现有脚本），生产可选 PostgreSQL 提升并发与可靠性。
  - 部署支持版本化构建与回滚（见 deployment 文档），与「内容版本回滚」配合使用。
  - 长任务代理每轮实现单一 feature，且需通过既定验证命令（如 `pnpm lint && pnpm type-check`）及核心链路回归。

## 6. 验收标准

- **可量化验收项**：
  - 所有目标页面（home、about、advantages、core-services、cases、experts、contact、global）均可通过后台编辑并发布到前台，且前台按 config + locale 正确展示。
  - 发布门禁：存在必填字段 missing 或 stale 时，发布被拒绝并返回明确错误信息。
  - 回滚：回滚到指定历史版本后生成新发布版本，公共接口立即可读且内容与所选版本一致。
  - 公共接口仅返回发布态，在任何情况下不返回草稿或未发布配置。
  - 草稿并发：双用户同时编辑同一页面并保存时，后提交者收到 409，刷新后基于最新版本可继续编辑。
  - 至少通过 lint + type-check，且核心链路（登录 → 编辑草稿 → 校验 → 发布 → 前台拉取；回滚 → 前台拉取）具备回归验证（自动化或手动检查清单）。
  - 角色权限：editor 无法调用发布/回滚接口（403）；admin 可完整走通发布与回滚流程。

## 7. 风险与依赖项说明

- **依赖项**：实现依赖 `docs/architecture.md`、`docs/api-spec.md`、`docs/data-model.md` 及 `docs/development-plan.md` 的约定；运行依赖 Node.js 20+、pnpm、Go 1.21+ 及开发/生产数据库（SQLite 或 PostgreSQL）。业务验收以当前 8 个 pageKey 的 schema 与双语门禁规则为准。
- **风险**：开发环境使用 SQLite，多用户同时编辑或高并发发布时存在锁竞争风险，生产建议评估并切换 PostgreSQL。Schema 与双语状态（missing/stale/done）需在后台校验、发布门禁与前台消费间保持一致，任一处单独变更易导致门禁失效或前台异常；实施细节与分阶段风险见 `docs/development-plan.md` 第 7 节。
